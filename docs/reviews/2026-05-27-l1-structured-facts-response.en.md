## Executive summary

The expert review filed at
[`docs/reviews/2026-05-27-l1-structured-facts-for-ai-unbiased-understanding.en.md`](2026-05-27-l1-structured-facts-for-ai-unbiased-understanding.en.md)
argues that the L1 Feature Registry (W1-W5 shipped 2026-05-27, ADR-0151 /
ADR-0152, Rule G-14 now blocking) is moving in the right direction but
still falls short of an **AI-unbiased understanding** standard. The
expert's stricter requirement, paraphrased:

> Structured facts used by AI for implementation decisions MUST be
> generated by deterministic scripts from authoritative sources (code,
> contracts, tests, build metadata, runtime configuration, ADRs). LLMs
> may explain or critique, but MUST NOT author or refresh the structured
> fact layer.

**Engineering adopts this hard rule.** It is operationalised as a new
fact layer under `architecture/facts/`, a six-wave PR train, ADR-0154,
Rule G-15 / enforcer E179 / gate Rule 131, and a sunset roadmap for
hand-authored factual fields. The bounded scope of v1 is: cover the
**factual** surface (code, contracts, tests, build, runtime config, ADR)
without removing the **intent** surface (synopsis, goals, non-goals,
AI Execution Boundary, ADR rationale, L1 prose). Hand-authored prose
remains canonical for intent / rationale / trade-offs / non-goals; it
loses authority only for factual claims that overlap with extracted
facts.

## Push-backs (technical rebuttal of expert wording)

Engineering pushed back on five aspects of the expert's wording; four
were accepted, one was overruled by the user on 2026-05-27.

### Push-back 1 (accepted): SFR-3 "JavaParser wherever possible" → ASM-primary hybrid

The expert recommended JavaParser as the source-AST extractor. Rebuttal:

1. JavaParser parses **source** and misses annotation-processed types
   (Lombok, MapStruct, springdoc-openapi descriptors).
2. The authoritative surface for "what does the production runtime
   actually expose?" is **compiled bytecode** under `target/classes`.
   ASM is the right tool there — small, fast, and deterministic.
3. JavaParser still wins where the compiler erases information that
   matters (Javadoc, original comments, source line spans).

**Resolution:** Wave 4 uses ASM as the primary extractor over
`target/classes`; JavaParser overlays Javadoc and source spans onto the
ASM facts. Both live inside `tools/architecture-workspace` (already a
Java module with Maven wiring).

### Push-back 2 (accepted): SFR-2 `generated_at_utc` is reproducibility-hostile

The expert's provenance field list included `generated_at_utc`. Rebuttal:
a live timestamp would tick on every extraction and break Rule G-13.b's
byte-identical-on-regen contract. Two extractor runs at the same
workspace HEAD would diff.

**Resolution:** `generated_at_utc` is INTENTIONALLY OMITTED from the
fact schema (see `architecture/facts/schema/fact.schema.yaml`).
`repo_commit` (40-char lowercase git SHA-1 of the workspace HEAD at
extraction time) is the SLSA-style provenance anchor and is naturally
stable across reruns at the same HEAD.

### Push-back 3 (accepted): SFR-8 drift gates — extend Rule G-13, do not duplicate

The expert recommended "adding CI gates that enforce byte-identical
regeneration for generated fact files, generated DSL fragments,
normalized architecture model, rendered L1 feature catalogs, rendered
SPI appendices, rendered development views." Rebuttal: Rule G-13
(Single-Source Rendering Coherence) already polices rendered surfaces.
A parallel rule would duplicate G-13.b and create a kernel-card
coherence defect (Rule G-3 polices this).

**Resolution:** Rule G-15 covers only the fact-layer–specific contract
(provenance fields, banner presence, LLM-no-author). At W6 of this
plan, `docs/governance/templates/surface-classification.yaml` adds rows
for `architecture/facts/generated/*.json` classifying them as
`templated` with the extractor as the renderer — so G-13.b ALSO catches
fact-layer drift as defence-in-depth.

### Push-back 4 (accepted): SFR-9 "Add an explicit rule" → sub-clause of Rule G-15

The expert recommended adding "an explicit rule" forbidding LLM-
authored fact refreshes. Rebuttal: the discipline is already implicit
in Rule G-1.b ("generated fragments... never hand-edited, and byte-
identical on regeneration"). Adding a fresh top-level rule for the
LLM aspect would create near-duplicate kernels (Rule G-3 would block
the duplication).

**Resolution:** the LLM-no-author discipline is codified as Rule G-15
sub-clause .c. No fresh top-level rule.

### Push-back 5 (overruled by user 2026-05-27): SFR-5 FunctionPoint schema size

Engineering initially proposed a v1/v2 split for the FunctionPoint
schema: 9 fields in v1 (fp_id, feature_id, name, actor, channel,
trigger, code_entrypoint_refs, test_refs, contract_op_refs), 5 more
in v2 after a soak window. Rationale: with 15 FPs × 14 fields = 210
cells, many of which would start `null`, the gate would emit noisy
warnings on Wave 5 ship.

**User counter-decision 2026-05-27:** accepted the expert's full
14-field schema in v1. Mitigation accepted: Rule G-15 sub-clause .d
blocks ONLY on the three hard-evidence fields (`code_entrypoint_refs`,
`test_refs`, and at least one of
`input_contract_refs` / `output_contract_refs` for `http`/`spi`
channel FPs). The remaining 11 fields are advisory: `null` is
acceptable until a source authority exists, but a non-null value MUST
resolve back to a fact id OR a named ADR / contract / policy file.

This counter-decision is recorded for traceability; it does not change
the per-SFR decision (SFR-5 is `Accept (full)` in the table above).

## Wave roadmap (pre-committed contiguous PR train)

User decision 2026-05-27: pre-committed contiguous PR train. All six
waves open as back-to-back branches under one continuous tag sequence
(`v2.1.0-fact-layer-rc1` ... `rc6`); no per-wave re-authorization
between waves once Wave 1 lands. The train is stoppable mid-flight only
by explicit user intervention.

### Wave 1 — Fact schema, provenance, ADR-0154, response file (this PR)

Adds `architecture/facts/{README.md, schema/fact.schema.yaml,
generated/.gitkeep}`,
`architecture/profile/saa-property-authority.yaml`, Rule G-15 kernel +
card + gate Rule 131 advisory, gate/lib/check_fact_layer_integrity.py,
ADR-0154, this response file. Extends Rule G-13 kernel/card to name
`architecture/facts/generated/*` as a recognized render-idempotent
surface. Updates `docs/governance/architecture-status.yaml` baselines +
`allowed_claim`; updates `docs/governance/recurring-defect-families.yaml`
with new family F-hand-authored-factual-drift; adds E179 row to
`docs/governance/enforcers.yaml`.

### Wave 2 — Module / build / ADR / runtime-config extractors

Adds `tools/architecture-workspace/.../facts/` package with
`ModuleBuildFactExtractor`, `AdrFactExtractor`, and
`RuntimeConfigFactExtractor`. Each reads already-structured YAML
sources (pom.xml, module-metadata.yaml, dfx YAMLs, docs/adr/*.yaml,
application*.yaml + `@ConfigurationProperties`). Wires Maven goal
`mvn -pl tools/architecture-workspace exec:java@extract-facts`. Rule
G-15.b becomes blocking on provenance field presence for the three
emitted files. Re-runs and confirms the drift-check noted by the
expert (`render_features_catalog.py --check`).

### Wave 3 — Contract extractor + rendered contract catalog

Adds `ContractFactExtractor` reading
`docs/contracts/openapi-v1.yaml` + `docs/contracts/*.v1.yaml`, emits
`contract-surfaces.json`. Adds
`gate/lib/render_contract_catalog.py` + Jinja2 template
`docs/governance/templates/contract-catalog.md.j2` consuming the
facts. `docs/contracts/contract-catalog.md` becomes rendered. A
function point claiming an operation that does not appear in
`contract-surfaces.json` fails Rule G-15.

### Wave 4 — Code-symbol + test inventory extractors (ASM + JavaParser hybrid)

Adds `CodeSymbolFactExtractor` (ASM over `target/classes` + JavaParser
overlay for Javadoc and source spans) and `TestInventoryFactExtractor`
(ASM over `target/test-classes` + Surefire/Failsafe XML report
parsing). Cross-check enforcer: every entry in
`module-metadata.yaml#spi_packages` MUST appear in `code-symbols.json`
with ≥1 public interface; every `saa.verificationTestFqns` entry MUST
resolve to a test class in `tests.json`. Rule G-15.c (byte-identical
regen + LLM-no-author banner) becomes blocking.

### Wave 5 — Thicker FunctionPoint (full 14-field schema) + seed mount

Extends `architecture/features/function-points.dsl` to the full 14-
field schema:

- factual (hard-evidence, blocking): `code_entrypoint_refs[]`,
  `test_refs[]`, `input_contract_refs[]`, `output_contract_refs[]`;
- intent (advisory): `actor`, `channel`, `trigger`, `preconditions`,
  `error_modes`, `state_changes`, `side_effects`,
  `auth_policy_refs`, `tenant_policy_refs`,
  `idempotency_policy_refs`, `observability_refs`,
  `source_authority_refs`.

One-shot `FunctionPointSeedMounter` parses the existing 7-column prose
tables under `architecture/docs/L1/agent-service/features/*.md` and
emits a draft patch for human review (same pattern as the W2
`capabilities.dsl` initial mount). Implements `contained_by` and
`decided_by` edges that the W1 schema declared but left at zero edges.

### Wave 6 — AI consumption contract + sunset grandfathered fields

Updates `AGENTS.md` and `architecture/README.md` reading order to
place `architecture/facts/generated/` BEFORE prose for factual
claims. Updates CLAUDE.md auto-memory contract guidance pointing AI
agents at the fact layer. Walks
`architecture/profile/saa-property-authority.yaml` sunset list and
retires `saa.devPaths`, `saa.verificationTestFqns`,
`saa.verificationCommands`, and `saa.sourceFile` from authored →
generated. Promotes the L1-doc back-reference check from advisory to
blocking (after a 14-day soak).

## Non-goals (mirrored from review + adjusted)

- **Do not** remove ADRs or L1 prose. They remain canonical for
  intent.
- **Do not** force every architecture concept into Structurizr
  scalar properties. Rich payloads live in JSON facts.
- **Do not** allow an LLM to fabricate missing facts to make the
  graph look complete. Missing extraction = missing fact, until the
  extractor or source authority catches up.
- **Do not** introduce competing rules; extend Rule G-13 and add
  Rule G-15 only. Rule G-3 polices kernel-card coherence and will
  block fresh near-duplicate rules.
- **Do not** rely on `generated_at_utc` for provenance — use
  `repo_commit` (SLSA-style stable across reruns).
