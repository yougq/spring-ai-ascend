# Fact-Layer Delivery Second Correction Request

Engineering team,

The Round-2 correction moved in the right direction: the shipped HTTP
FunctionPoint references now resolve against generated code, test, and
contract facts; the removed `FP-LIST-RUNS` surface no longer appears as an
active FunctionPoint; direct `ExtractFactsCli --check`, the Python fact-layer
integrity checker, the feature catalog check, the gate self-tests, and
`./mvnw -Pquality verify` all pass in the reviewed workspace.

However, the delivery cannot be accepted yet. The remaining issues are not
cosmetic. They affect whether the fact layer is actually trustworthy as the
AI's unbiased L1 input. Our requirement is explicit: structured facts must be
derived from real authoritative sources by deterministic scripts, not authored
or refreshed by an LLM or by manual edits. The canonical gate must therefore
fail closed when generated facts drift from extractor output.

## Non-Negotiable Acceptance Standard

Rule G-15 and ADR-0154 are the acceptance authority for this work:

- `docs/adr/0154-fact-layer-authority.yaml:2` names the fact layer as
  "generated structured facts as the AI's primary L1 input".
- `docs/adr/0154-fact-layer-authority.yaml:50-51` states that L1 structured
  facts used for AI implementation guidance must be generated by deterministic
  scripts from authoritative source surfaces.
- `CLAUDE.md:407-409` defines Rule G-15 and states that generated structured
  facts live under `architecture/facts/generated/`, are produced only by
  deterministic extractor binaries, and must be byte-identical to extractor
  re-emission at the same workspace HEAD.
- `architecture/facts/README.md:36-39` says `generated/` is owned by
  deterministic extractor scripts only, with the LLM no-author rule.
- `architecture/facts/README.md:149-152` marks G-15.b, G-15.c, and G-15.d
  as blocking, including the `ExtractFactsCli --check` byte-diff.

The remaining requests below are the minimum required to make the current
implementation match those authorities.

## Requirement 1 - Make `ExtractFactsCli --check` genuinely blocking in the canonical gate

### Problem

The canonical gate currently claims to wire `ExtractFactsCli --check` as a
blocking byte-diff, but the shell implementation swallows the extractor
failure:

- `gate/check_architecture_sync.sh:7084-7096` documents the Round-2 Wave B
  Java byte-diff gate.
- `gate/check_architecture_sync.sh:7092` runs:
  `... exec:java@extract-facts ... --check" 2>&1 || true`.
- `gate/check_architecture_sync.sh:7093` then captures `$?`, but because the
  command substitution ends with `|| true`, the captured return code is always
  `0`.

This makes the gate fail-open for the exact drift scenario G-15.c is supposed
to block.

### Sources supporting the requirement

- `CLAUDE.md:409`: G-15.c requires files under `architecture/facts/generated/`
  to be byte-identical to extractor re-emission at the same workspace HEAD.
- `docs/governance/rules/rule-G-15.md:88-102`: the rule card defines the
  byte-identical regeneration and LLM no-author obligation.
- `architecture/facts/README.md:149-152`: the lifecycle table marks
  `ExtractFactsCli --check` byte-diff as blocking.
- `gate/check_architecture_sync.sh:7092-7096`: the current implementation
  attempts the check but loses the real return code.
- `gate/test_architecture_sync_gate.sh:7129-7214`: current Rule 131 self-tests
  cover the FunctionPoint resolver and structure-positive path, but do not
  contain a G-15.c negative fixture that mutates a generated fact file and
  proves the canonical gate fails.

### Required change

Replace the `|| true` return-code pattern with a real fail-closed capture.
For example:

```bash
if ! _r131_extract_out=$(./mvnw -f tools/architecture-workspace/pom.xml -q exec:java@extract-facts -Dexec.args="--repo $PWD --out architecture/facts/generated --check" 2>&1); then
  _r131_first=$(printf '%s' "$_r131_extract_out" | tail -3 | head -1)
  fail_rule "fact_layer_integrity" "ExtractFactsCli --check drift: ${_r131_first:-extractor failed} -- Rule G-15.c (Java byte-diff) / E179"
  _r131_fail=1
fi
```

If compiled classes are required for the extractor, the default canonical
verification path must still fail closed with a clear remediation message when
the required class directories are missing. A non-blocking or advisory mode may
exist only as an explicit opt-in mode, not as the default acceptance path.

### Acceptance evidence required

- Add a negative self-test that mutates at least one file under
  `architecture/facts/generated/` and proves the canonical gate reports
  `ExtractFactsCli --check drift`.
- Show `bash gate/test_architecture_sync_gate.sh` passing with that new
  negative case included in the test count.
- Show `bash gate/check_architecture_sync.sh` passing on clean generated facts.
- Show the same gate failing after an intentional generated-fact mutation,
  then passing again after regenerating through `ExtractFactsCli`.

## Requirement 5 - Align `TestInventoryFactExtractor` documentation with implementation

### Problem

`TestInventoryFactExtractor` claims more than it currently does:

- `tools/architecture-workspace/src/main/java/com/huawei/ascend/tools/architecture/facts/TestInventoryFactExtractor.java:29-30`
  says Surefire/Failsafe XML reports are inspected to attach a
  `recent_run_present` flag.
- A repository search for `recent_run_present`, `surefire`, `failsafe`, and
  XML parsing in the extractor and generated `tests.json` only finds that
  comment, not an implementation or generated field.

This is lower severity than the gate fail-open issue, but it matters for AI
understanding: a fact extractor must not imply that a test was recently
executed unless the generated fact actually carries that evidence.

### Sources supporting the requirement

- `tools/architecture-workspace/src/main/java/com/huawei/ascend/tools/architecture/facts/TestInventoryFactExtractor.java:24-30`.
- `architecture/facts/generated/tests.json`: no `recent_run_present` field was
  found during review.
- Rule G-15.b provenance requirements in `CLAUDE.md:409` require generated
  facts to be precise about what was actually observed.

### Required change

Choose one of these two paths:

1. implement Surefire/Failsafe XML report parsing and emit a schema-covered
   `recent_run_present` or equivalent field, with tests; or
2. remove the overclaiming Javadoc and state that the current extractor only
   identifies JUnit-bearing compiled test classes under `target/test-classes`.

### Acceptance evidence required

- If implemented: show generated `tests.json` entries with the new field, the
  schema update, and unit tests covering report-present and report-absent
  cases.
- If deferred: show the Javadoc correction and explicitly track XML-based
  recent-run evidence as a future enhancement, not a shipped fact.
