# Recommendation: Script-Generated Structured Facts for AI-Unbiased L1 Understanding

Date: 2026-05-27

Audience: Engineering team maintaining the Spring AI Ascend architecture workspace,
L1 feature registry, contract catalog, gates, and generated projections.

## Executive summary

The current L1 Feature Registry is moving in the right direction: it introduces
typed `SAA Feature` elements, function points, lifecycle states, verification
metadata, rendered per-module catalogs, and an AI Execution Boundary. However,
it still does not meet the stronger requirement for AI-unbiased understanding:
AI agents must not have to infer facts from human prose, hand-authored DSL
properties, or stale rendered Markdown.

The required direction is stricter:

> Structured facts used by AI for implementation decisions MUST be generated by
> deterministic scripts from authoritative sources: source code, contracts,
> tests, build metadata, runtime configuration, and ADRs. LLMs may explain,
> critique, or propose changes, but MUST NOT author or refresh the structured
> fact layer.

Human-authored architecture documents remain important, but they should carry
intent, rationale, trade-offs, risks, and non-goals. They should not be the
primary authority for factual claims about code shape, API contracts, SPI
signatures, test coverage, module dependencies, runtime entrypoints, or
verification commands.

## Current root cause

Root cause: the L1 Feature Registry currently treats hand-authored DSL and
rendered Markdown as the structured source for several facts that should be
extracted from code, contracts, tests, build metadata, and configuration.

Local evidence:

- `architecture/features/README.md` says the feature directory is the
  "canonical authored source" for L1 feature inventory and marks
  `capabilities.dsl`, `function-points.dsl`, `features.dsl`, and
  `verification.dsl` as `AUTHORED`.
  Source: [`architecture/features/README.md`](../../architecture/features/README.md)
- `features.dsl` requires hand-authored `saa.synopsis`, `saa.devPaths`,
  `saa.verificationTestFqns`, and `saa.verificationCommands`.
  Source: [`architecture/features/features.dsl`](../../architecture/features/features.dsl)
- `function-points.dsl` currently gives each function point only a short
  description plus common metadata such as owner/status/source ADR. It does not
  structurally carry actor, trigger, preconditions, input/output schemas,
  state changes, error modes, idempotency policy, contract operation, or code
  entrypoints.
  Source: [`architecture/features/function-points.dsl`](../../architecture/features/function-points.dsl)
- The relationship vocabulary defines `contained_by` and `decided_by`, but the
  current `features.dsl` only contains function-point `contains` relationships
  in the feature section; no actual `contained_by` or `decided_by` relationship
  was found in that file during this review.
  Sources: [`architecture/profile/relationship-types.yaml`](../../architecture/profile/relationship-types.yaml),
  [`architecture/features/features.dsl`](../../architecture/features/features.dsl)
- The generated catalog is not currently byte-identical for all modules:
  `python gate/lib/render_features_catalog.py --check` reported drift for
  `architecture/docs/L1/agent-service/features/README.md` during this review.
  Source: local command output from the review session.

## Principle

The architecture workspace should become a structured index over generated
facts, not the hand-authored origin of all facts.

Recommended authority split:

| Layer | Source | Allowed author | Purpose |
|---|---|---|---|
| Fact layer | `architecture/facts/generated/*.json` or `.yaml` | deterministic scripts only | Extracted facts from code, contracts, tests, build files, config |
| Architecture index | `architecture/workspace.dsl` and projected DSL fragments | scripts plus tightly reviewed human mapping where necessary | Identity, hierarchy, typed relationships, views |
| Rationale layer | ADRs and L1 narrative docs | humans, with review | Intent, trade-offs, non-goals, risks, future direction |
| AI consumption layer | generated fact graph / normalized model | scripts only | Queryable, source-backed inputs for AI agents |

## Requirements

### SFR-1: Introduce a generated structured fact layer

Required change:

Create a generated fact layer under `architecture/facts/generated/` that is
produced only by deterministic scripts. The first version should include at
least:

- `code-symbols.json`: Java packages, public interfaces, records, enums,
  annotations, Spring controllers, configuration classes, and method signatures.
- `contract-surfaces.json`: OpenAPI operations, contract YAML files, schemas,
  status labels, request/response/error shapes, and runtime-enforced flags.
- `tests.json`: JUnit, Failsafe, ArchUnit, and gate tests with source files,
  test class names, likely target symbols when extractable, and verification
  commands.
- `module-build.json`: Maven modules, dependencies, module metadata, SPI
  package declarations, and allowed dependency direction.
- `runtime-config.json`: Spring configuration properties, auto-configuration
  classes, conditional beans, profile/posture guards, and default values.

The files MUST be generated by scripts and MUST carry a generated-file banner.
LLMs MUST NOT edit them directly.

Local evidence:

- The current architecture workspace already treats `generated/` as a machine
  zone and says generated fragments are never hand-edited.
  Source: [`architecture/README.md`](../../architecture/README.md)
- The current feature inventory is still hand-authored even though it contains
  facts about dev paths, test FQNs, and verification commands.
  Source: [`architecture/features/README.md`](../../architecture/features/README.md)

External support:

- Structurizr DSL is explicitly a text-based way to define a software
  architecture model, which supports the repository's model-as-code direction:
  <https://docs.structurizr.com/dsl>
- Structurizr describes itself as a "models as code" tool for C4, with DSL
  rendered into diagrams:
  <https://docs.structurizr.com/as-code>
- Reproducible Builds defines reproducibility as recreating identical artifacts
  from the same source, environment, and instructions. The same principle should
  apply to generated architecture facts:
  <https://reproducible-builds.org/docs/definition/>
- SLSA provenance defines provenance as verifiable information about where,
  when, and how an artifact was produced:
  <https://slsa.dev/provenance/>

Acceptance criteria:

- `architecture/facts/generated/` exists.
- Every generated fact file is recreated by a documented command.
- Re-running the generators without source changes produces byte-identical
  output.
- CI fails if a generated fact file is hand-edited or stale.

### SFR-2: Make generated facts provenance-bearing

Required change:

Every structured fact MUST include provenance sufficient for an AI agent or
reviewer to trace it back to the source of truth.

Minimum required fields:

```yaml
fact_id: string
fact_kind: code_symbol | contract_operation | schema | test | build_module | runtime_config | relationship
source_kind: code | contract | test | build | config | adr
source_path: string
source_symbol: string
source_span: string
extractor: string
extractor_version: string
repo_commit: string
observed_value: object
generated_at_utc: string
```

If a fact is derived from multiple sources, it MUST include `source_refs[]`.
For example, a "feature verifies function point" fact may depend on a Java test
class, the generated test inventory, the function point ID, and a gate rule.

Local evidence:

- `verification.dsl` currently maps tests to function points by hand-authored
  `SAA Test` elements and `verifies` relationships.
  Source: [`architecture/features/verification.dsl`](../../architecture/features/verification.dsl)
- The contract catalog already maps SPI and runtime surfaces to owners and
  packages, which can seed generated cross-checks.
  Source: [`docs/contracts/contract-catalog.md`](../contracts/contract-catalog.md)

External support:

- SLSA provenance tracks artifacts back to source code and describes where,
  when, and how something was produced:
  <https://slsa.dev/provenance/>
- Reproducible Builds emphasizes source revision, environment, build
  instructions, and bit-for-bit verification:
  <https://reproducible-builds.org/docs/definition/>

Acceptance criteria:

- Every generated fact has `source_path`, `source_kind`, `extractor`, and
  `repo_commit`.
- The fact generator records its own version or class name.
- CI can identify which source file must change when a fact is wrong.

### SFR-3: Generate code facts from source code, not prose

Required change:

Build a code fact extractor for Java source under all Maven modules. It should
use a parser or compiler-backed model rather than regular expressions wherever
possible.

Minimum extracted facts:

- public interfaces and their method signatures
- public records and record components
- public enums and enum values
- Spring controllers and route annotations
- configuration properties and auto-configuration classes
- SPI packages and package-level metadata
- production package tree mapped to module and layer
- test classes and test method names

Local evidence:

- The repository already has real Java SPI surfaces such as `RunRepository`,
  `EngineRegistry`, `S2cCallbackTransport`, `ExecutorAdapter`, `ModelGateway`,
  and many records/enums. These should be extracted from code rather than
  copied into prose.
  Sources:
  [`agent-service/src/main/java`](../../agent-service/src/main/java),
  [`agent-bus/src/main/java`](../../agent-bus/src/main/java),
  [`agent-execution-engine/src/main/java`](../../agent-execution-engine/src/main/java),
  [`agent-middleware/src/main/java`](../../agent-middleware/src/main/java)
- L1 development docs map layers to packages, but those mappings should be
  validated against the real package tree.
  Source: [`architecture/docs/L1/agent-service/development.md`](../../architecture/docs/L1/agent-service/development.md)

External support:

- JavaParser documentation shows parsing Java code into an AST and inspecting
  AST structure, including YAML-like structure output:
  <https://javaparser.org/inspecting-an-ast/>
- C4 uses hierarchical abstractions down to code, which supports connecting
  architecture views to real implementation structure:
  <https://c4model.com/>

Acceptance criteria:

- A command such as `java ... CodeFactExtractor --repo . --out architecture/facts/generated/code-symbols.json`
  or an equivalent Maven goal exists.
- The extractor output includes symbol-level facts, not only file paths.
- L1 docs and workspace projections consume these generated facts instead of
  manually listing source paths where possible.

### SFR-4: Generate contract facts from machine-readable contracts

Required change:

Extract contract facts from OpenAPI and YAML contract files, then link those
facts to modules, features, function points, and tests.

Minimum extracted facts:

- HTTP method and path
- operation ID or generated operation key
- request schema references
- response schema references
- error envelope codes
- auth/tenant/idempotency headers
- contract status: `design_only`, `schema_shipped`, `runtime_enforced`
- source ADRs and enforcing tests when declared

Local evidence:

- The current rendered feature catalog has a "Contract Surface" section, but it
  only points readers to the general contract catalog instead of listing
  feature-level contract facts.
  Source: [`architecture/docs/L1/agent-service/features/README.md`](../../architecture/docs/L1/agent-service/features/README.md)
- The contract catalog already contains runtime promise surfaces such as
  OpenAPI, engine envelope, hooks, and S2C callback.
  Source: [`docs/contracts/contract-catalog.md`](../contracts/contract-catalog.md)

External support:

- OpenAPI defines a standard interface description for REST APIs that humans
  and computers can use to understand a service without source code,
  additional documentation, or network traffic inspection:
  <https://www.openapis.org/faq>

Acceptance criteria:

- Generated contract facts enumerate operation-level details.
- Each function point that corresponds to an API operation references a
  generated contract fact ID.
- CI fails when a shipped function point claims an operation that does not exist
  in OpenAPI or the contract catalog.

### SFR-5: Replace thin FunctionPoint metadata with generated, development-ready facts

Required change:

Extend the FunctionPoint model so it is directly useful for AI-assisted
development. The target shape should be generated or mechanically validated
from source facts wherever possible.

Recommended FunctionPoint schema:

```yaml
fp_id: string
feature_id: string
name: string
actor: string
channel: http | spi | event | scheduled | internal
trigger: string
preconditions: list
input_contract_refs: list
output_contract_refs: list
error_modes: list
state_changes: list
side_effects: list
auth_policy_refs: list
tenant_policy_refs: list
idempotency_policy_refs: list
code_entrypoint_refs: list
test_refs: list
observability_refs: list
source_authority_refs: list
```

Local evidence:

- Current function points have a useful seed inventory but lack the fields
  above.
  Source: [`architecture/features/function-points.dsl`](../../architecture/features/function-points.dsl)
- The `agent-service` deep-dive markdown files already contain richer function
  point columns such as covered clusters, inputs/outputs, collaborators,
  exception coverage, and OSS reference. These columns should be elevated into
  structured facts rather than remaining only in prose tables.
  Source: [`architecture/docs/L1/agent-service/features/access-layer.md`](../../architecture/docs/L1/agent-service/features/access-layer.md)

External support:

- IEEE/ISO/IEC 29148 defines requirements engineering processes, attributes,
  and characteristics for requirements throughout the lifecycle:
  <https://standards.ieee.org/ieee/802.1Q/6937/>
- INCOSE requirement guidance emphasizes properties such as unambiguous,
  verifiable, necessary, feasible, singular, complete, consistent, and
  traceable:
  <https://www.incose.org/docs/default-source/working-groups/requirements-wg/shared_gtwr/gtwr_characteristics_section_4_050423.pdf>
- arc42 includes context/scope, runtime view, deployment view, decisions,
  quality requirements, risks, and glossary as standard architecture
  documentation concerns:
  <https://arc42.org/overview>

Acceptance criteria:

- Function points expose enough structured facts for an AI agent to identify
  where to edit, which contract it must preserve, which tests to run, and which
  failure modes to consider.
- Shipped function points must have code entrypoint refs, contract refs where
  applicable, and test refs.
- No function point should rely on a one-line prose description as its only
  development guidance.

### SFR-6: Separate machine facts from human intent

Required change:

Keep human-authored material, but explicitly classify it as intent, rationale,
or governance. It must cite generated fact IDs when it talks about code,
contracts, tests, or runtime behavior.

Human-authored fields that may remain:

- intent
- goals
- non-goals
- trade-offs
- risk analysis
- design alternatives
- ADR rationale
- future direction
- glossary / ubiquitous language

Fields that should be generated or mechanically checked:

- source paths
- Java symbols
- SPI signatures
- OpenAPI routes
- contract schemas
- test class names
- verification commands
- module dependencies
- runtime configuration keys
- code-to-layer mappings

Local evidence:

- The current feature schema mixes human-intent fields with factual fields in
  `features.dsl`.
  Source: [`architecture/features/features.dsl`](../../architecture/features/features.dsl)
- The current L1 module template includes authored architecture narrative and
  rendered outputs in one directory shape. That is fine, but the source
  authority for each field must be explicit.
  Source: [`architecture/docs/L1/_template/README.md`](../../architecture/docs/L1/_template/README.md)

External support:

- arc42 separates architecture concerns such as decisions, risks, quality
  requirements, runtime view, and glossary:
  <https://arc42.org/overview>
- ISO/IEC 25010 defines a quality model usable for requirements, validation,
  testing objectives, quality control criteria, and acceptance criteria:
  <https://www.iso.org/cms/%20render/live/en/sites/isoorg/contents/data/standard/07/81/78176.html>

Acceptance criteria:

- Generated fact files never contain unreviewed design intent.
- Human-authored docs cite generated fact IDs for factual claims.
- Gates reject factual claims in L1 docs that conflict with generated facts.

### SFR-7: Make the architecture workspace consume generated facts

Required change:

Update `workspace.dsl` and generated DSL fragments so they consume generated
fact files, not hand-authored factual lists, wherever possible.

Recommended flow:

```text
source code / contracts / tests / build files / config
        -> deterministic fact extractors
        -> architecture/facts/generated/*.json
        -> deterministic DSL fragment emitters
        -> architecture/workspace.dsl closure
        -> deterministic Markdown renderers
        -> architecture/docs/L1/<module>/features/README.md
```

Local evidence:

- `architecture/workspace.dsl` already includes authored feature fragments and
  generated fragments.
  Source: [`architecture/workspace.dsl`](../../architecture/workspace.dsl)
- `architecture/README.md` already distinguishes authored and generated zones,
  and states that generated fragments are emitted by tooling.
  Source: [`architecture/README.md`](../../architecture/README.md)

External support:

- Structurizr DSL is designed to define a software architecture model in text:
  <https://docs.structurizr.com/dsl>
- C4 encourages hierarchical architecture abstractions and views, including
  context, containers, components, code, dynamic, and deployment diagrams:
  <https://c4model.com/>

Acceptance criteria:

- Generated fact files feed generated DSL fragments.
- Workspace validation includes fact-to-DSL consistency checks.
- Rendered Markdown is always downstream of generated facts, never a parallel
  source of truth.

### SFR-8: Add drift gates for facts, projections, and rendered catalogs

Required change:

Add CI gates that enforce byte-identical regeneration for:

- generated fact files
- generated DSL fragments
- normalized architecture model
- rendered L1 feature catalogs
- rendered SPI appendices
- rendered development views

The gate must fail if any generated artifact differs from the output of its
generator.

Local evidence:

- Rule G-13 already requires single-source rendering coherence for dynamic
  claims.
  Source: [`CLAUDE.md`](../../CLAUDE.md)
- `render_features_catalog.py --check` already detects catalog drift; during
  this review it reported drift for the agent-service feature catalog.
  Source: [`gate/lib/render_features_catalog.py`](../../gate/lib/render_features_catalog.py)

External support:

- Reproducible Builds requires that the same inputs and instructions recreate
  identical artifacts:
  <https://reproducible-builds.org/docs/definition/>
- SLSA provenance supports tracing artifacts to where, when, and how they were
  produced:
  <https://slsa.dev/provenance/>

Acceptance criteria:

- A single verification command can regenerate and compare all generated
  architecture artifacts.
- CI fails on generated-file drift.
- Generated-file drift is treated as an architecture correctness failure, not a
  formatting issue.

### SFR-9: Forbid LLM-authored fact refreshes

Required change:

Add an explicit rule:

> LLMs may propose changes to extractors, schemas, ADRs, and human-authored
> rationale. LLMs MUST NOT directly author or refresh generated structured
> facts, generated DSL fragments, or rendered catalogs. All such artifacts must
> be produced by deterministic scripts and verified by gates.

Local evidence:

- The repository already forbids hand-editing generated architecture fragments.
  Source: [`architecture/README.md`](../../architecture/README.md)
- The current L1 feature catalog still leaves important factual fields in
  hand-authored DSL, so the rule needs to be widened from generated fragments to
  generated facts.
  Source: [`architecture/features/README.md`](../../architecture/features/README.md)

External support:

- Reproducible Builds and SLSA both emphasize verifiable generation provenance.
  Sources:
  <https://reproducible-builds.org/docs/definition/>,
  <https://slsa.dev/provenance/>

Acceptance criteria:

- Generated fact files include "DO NOT EDIT" banners.
- CI detects manual edits.
- Contributor docs tell AI agents to modify extractor code or source authority,
  then run generators, instead of editing generated facts directly.

### SFR-10: Define an AI consumption contract

Required change:

Define how AI agents are allowed to consume architecture facts. The contract
should require agents to:

1. Read generated facts before L1 prose for implementation decisions.
2. Cite fact IDs and source refs when claiming what code, contract, or test
   exists.
3. Treat human-authored prose as intent/rationale unless backed by generated
   facts.
4. Refuse to act on stale or drifted generated artifacts.
5. Run the feature-specific verification command derived from generated
   facts, not from model memory.

Local evidence:

- `architecture/README.md` currently says newcomers can reach an unbiased
  architecture picture by reading the workspace and related surfaces.
  Source: [`architecture/README.md`](../../architecture/README.md)
- Current reading order does not yet include generated fact files because they
  do not exist.

External support:

- OpenAPI's value proposition is that humans and computers can discover and
  understand service capabilities without inspecting source or network traffic:
  <https://www.openapis.org/faq>
- IEEE/ISO/IEC 29148 frames requirements as lifecycle information items with
  attributes and characteristics, supporting the need for explicit consumption
  semantics:
  <https://standards.ieee.org/ieee/802.1Q/6937/>

Acceptance criteria:

- `AGENTS.md`, `architecture/README.md`, or a new AI-consumption rule points AI
  agents to the generated fact layer.
- AI-facing docs explicitly say that generated facts outrank prose for factual
  claims.
- The L1 reading path is updated to include generated facts before rendered
  feature catalogs.

## Recommended implementation waves

### Wave 1: Fact schema and drift prevention

- Add `architecture/facts/README.md`.
- Define JSON Schema or YAML Schema for generated facts.
- Add generated-file banners and provenance fields.
- Add a no-op or minimal extractor for module/build facts.
- Add CI check for byte-identical regeneration.

### Wave 2: Code and test extractors

- Implement Java source extraction for public symbols, records, enums,
  controllers, configuration, and tests.
- Generate `code-symbols.json` and `tests.json`.
- Cross-check `module-metadata.yaml#spi_packages` against real public
  interfaces.

### Wave 3: Contract extractor

- Extract OpenAPI and contract YAML facts.
- Link function points to contract operations and schemas.
- Fail CI when shipped function points lack contract refs where applicable.

### Wave 4: Feature and function-point projection

- Generate or mechanically validate feature/function-point DSL fragments from
  facts.
- Populate function point details from generated code/contract/test facts.
- Replace prose-only "Contract Surface" and "Functional Decomposition" sections
  with concrete tables rendered from facts.

### Wave 5: AI consumption contract

- Update `AGENTS.md`, `architecture/README.md`, and relevant rule cards.
- Add "generated facts first" to the L1 reading path.
- Add gates that fail when AI-facing docs make factual claims not backed by
  generated facts.

## Non-goals

- Do not remove ADRs or L1 prose. They remain necessary for intent, rationale,
  risk, and trade-offs.
- Do not force every architecture concept into Structurizr scalar properties.
  Structurizr should index and visualize the model; richer fact payloads should
  live in JSON/YAML generated facts.
- Do not let an LLM infer missing facts to make the graph look complete. Missing
  extracted facts should remain missing until a source authority or extractor is
  added.

## Decision requested from engineering

Please decide whether to adopt the following hard rule:

> L1 structured facts used for AI implementation guidance must be generated by
> deterministic scripts from authoritative source surfaces. Human-authored docs
> may explain intent and rationale, but they must not be the sole source of
> factual claims about code, contracts, tests, dependencies, runtime behavior,
> or verification.

If accepted, the first engineering task should be to add the generated fact
schema and a minimal extractor pipeline, then update the L1 reading path so AI
agents consume generated facts before prose catalogs.
