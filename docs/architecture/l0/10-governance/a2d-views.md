# A2D 双视图模型

## 目的

A2D 的已接受产物必须同时服务两个消费视角，且不得各自维护事实。

## 两个视图

| 视图 | 面向对象 | 回答的问题 | 主要来源 |
|---|---|---|---|
| 交付视图 | 开发者、模块负责人、AI agent、harness 生成器 | 如何实现、如何拆任务、如何验证、完成标准是什么。 | AI-4 生成的场景、能力、模块、状态、契约、不变量、harness 和验证矩阵；AI-6 交付视图。 |
| 4+1 评审视图 | 架构负责人、评审者、业务负责人、模块负责人 | 架构是否成立、边界是否清楚、风险是否可接受、哪些问题需要裁决。 | AI-4 生成的 A2D 产物；AI-5 4+1 评审包。 |

## 核心约束

**两个视图不得各自维护事实。** 它们必须引用同一批 A2D 已接受产物：

- 场景（BA-* + technical）
- 能力图
- 模块责任卡
- 状态归属矩阵
- ICD / 契约草案
- 架构不变量
- Harness 设计
- 验证矩阵

关键设计项通常必须同时进入两个视图。如果只能进入其中一个，必须在对应产物中说明原因和风险。

## 视图验收口径

每个准备进入已接受状态的关键设计项，必须回答：

| 检查项 | 交付视图 | 4+1 评审视图 |
|---|---|---|
| 事实来源 | 能追溯到场景、能力、模块责任卡、状态归属矩阵、ICD、不变量或 harness。 | 能追溯到同一批 A2D 已接受产物，而不是另起一套描述。 |
| 可交付性 | 能生成开发切片、实现任务草案、DoD、测试或验证证据。 | 说明该设计对逻辑、开发、进程、物理、场景至少一个视图的影响。 |
| 可裁决性 | 明确 owner、输入、输出、依赖、阻塞项和验收标准。 | 明确变更级别、影响面、开放问题、冲突和需要人类裁决的问题。 |
| 可验证性 | 进入验证矩阵，或说明为什么当前不可验证。 | 让评审者能判断架构一致性、风险覆盖和后续基线条件。 |

## 最小 A2D 工作包

一个版本进入 AI 自动实现前，至少需要形成以下工作包：

| 工件 | 是否必需 | 归档位置 |
|---|---|---|
| 版本意图 | 必需 | `10-governance/version-intents/<version>.md` |
| 版本架构边界 | 必需 | `10-governance/architecture-envelopes/<version>.md` |
| 架构审核包 | H2 前必需 | `10-governance/review-packets/<version>.md` |
| 交付视图 | 进入实现前必需 | `10-governance/delivery-projections/<version>.md` |
| BA 场景或模块场景 | 涉及新场景时必需 | `02-scenarios/` |
| 能力映射 | 涉及能力变化时必需 | `01-capabilities/` |
| 模块责任卡 | 涉及模块职责变化时必需 | `04-modules/` |
| 状态归属 | 如涉及状态则必需 | `06-state/` |
| 契约草案 | 如涉及跨模块交互则必需 | `05-contracts/` |
| 模块设计包 | 模块并行开发必需 | `04-modules/<module>/` |
| Harness 设计 | 进入实现前必需 | `08-harness/` |
| 验证矩阵行 | 必需 | `09-verification/` |

## 版本级 Ready / Done 判定

| 阶段 | Ready | Done |
|---|---|---|
| 进入 AI 批量分析 | 版本意图已说明目标、范围、非目标、约束和发布门槛。 | 需求已归一化，架构影响矩阵和升级候选已形成。 |
| 进入 AI 批量建模 | H1 已确认版本架构边界。 | A2D 产物已按影响面批量生成或更新，开放问题有归宿。 |
| 进入人类架构审核 | 4+1 评审视图和交付视图已生成。 | H2 已确认架构包，例外项已裁决或进入后续位置。 |
| 进入自动实现 | 版本架构边界、交付视图、DoD、harness 计划和验证矩阵增量齐备。 | 代码变更、测试变更、实现证据完成。 |
| 进入发布例外审核 | 集成验证完成，漂移检查和风险说明已形成。 | H3 已确认剩余例外、发布风险和归档条件。 |
| 进入归档 | 无阻塞 finding，或阻塞项已明确移出本版本。 | 架构基线说明、替代关系记录、结转开放问题清楚。 |

## 内部活动 Ready / Done 判定

| 阶段 | Ready | Done |
|---|---|---|
| 进入场景建模 | A2D Intake Record 已说明用户、目标、约束。 | Business Activity Scenario 被业务和架构负责人确认。 |
| 进入能力拆解 | Business Activity Scenario 能串起模块。 | Capability Mapping 有 owner 或 Open Issue。 |
| 进入模块设计 | 模块责任卡已确认。 | 模块设计包可支撑任务拆解。 |
| 进入 harness | State Ownership、ICD、Scenario Assertions 明确。 | Harness Design 能覆盖核心状态、契约和失败路径。 |
| 进入实现 | 模块设计包、harness 设计、DoD 齐备。 | 代码变更、测试变更、实现证据完成。 |
| 进入归档 | 集成验证完成且无阻塞 finding。 | 架构基线说明、替代关系记录、结转开放问题清楚。 |
