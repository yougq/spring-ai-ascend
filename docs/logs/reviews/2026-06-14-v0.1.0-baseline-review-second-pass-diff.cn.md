## 二、增强:第二遍**新增 / 加深**的发现(第一遍没有或更浅)

### 维度 A(正确性/并发)
1. **🆕 超时被静默吞成"假 COMPLETED"(第二遍 H3)** — 第一遍只把 Versatile 超时当 MEDIUM"task 永久 WORKING"(A-M4);第二遍证明了**更坏的失败模式**:行间空闲超时 → `poll` 返回 null → 返回 `POISON` 且 `upstreamError==null` → 适配器当成**正常结束** → 调用方收到 **COMPLETED + 截断的半截答案**。对银行工作流这是"数据完整性"级问题,严重度应高于第一遍。
2. **🆕 AgentScope 出站 body 读取无空闲超时(第二遍 M1)** — 第一遍的超时分析只覆盖 Versatile;第二遍发现 AgentScope 的 SSE body 用 `ofLines().join()` **无读超时**,上游发完 header 后挂起会无限钉住一个 a2a-server 池线程。
3. **🆕 `ConcurrentHashMap.compute` 锁内做阻塞 close(第二遍 M5)** — `obtainTransport` 在 compute 重映射函数内调用 `transport.close()`(网络拆除),持 bin 锁期间阻塞所有哈希到同 bin 的并发调用。第一遍只说了"transport 不在生命周期内关"(A-M3),没看到这个锁内阻塞。
4. **🆕 `extractText` 全非文本入站 → 空 query 仍 COMPLETED(第二遍 L3)** — DataPart-only 消息被降级成空输入跑 agent,记日志但不拒绝 → 又一条"错误结果被报成成功"。
5. **🆕 `VersatileClient.getTimeout()` 为 null 时请求期 NPE(第二遍 L2)** — 构造期只给 connectTimeout 兜了 30s 默认,请求/poll 路径直接调 `getTimeout()`,null 时运行期才崩。

### 维度 B(安全)
6. **🆕 提交了内网拓扑 IP 作出厂默认(第二遍 MEDIUM)** — `VersatileProperties.java:37` 默认 `https://100.93.15.185:30001`、e2e 示例 `http://7.213.200.213:3001`(Huawei 内网段)。第一遍只抓了 `ssl-verify:false`(B-M2),没注意到**内网 IP 拓扑泄漏 + 忘改就明文打到内网旧主机**。
7. **🆕 无请求体大小上限 → 廉价 DoS(第二遍 LOW)** — 未授权大 body POST 直接全量解析,无 `maxInMemorySize` 限制。
8. **➕ 卡片投毒 SSRF 跳板(第二遍 MEDIUM,第一遍是 LOW)** — 第一遍 B-L1 只说"开放 schema + verbatim 转发"。第二遍**加深**:`RemoteAgentCardCache.resolveUrl` 对远程卡片里的**绝对 url 原样采信**,被攻陷/恶意的已信任远程可把出站 JSON-RPC 重定向到任意主机(卡片投毒→SSRF/外泄),并把远程 skill 描述原样喂给编排 LLM(注入放大)。证据链比第一遍完整得多。
9. **🆕 脱敏只按 key 名,自由文本值漏过(第二遍 LOW)** — `TrajectoryMasking` 只在 key 命中正则时脱敏 value,用户消息正文(benign key 下的客户姓名等)只截断不脱敏。银行数据分类视角的真实残留。
10. **➕ 日志违规证据更硬** — 第二遍指出 INFO 打原始载荷**直接违反该 package 自己 `package-info.java` 写明的脱敏约定**,并多列了 4 处违规点(`A2aResultRouter:66/83/121`、`A2aParentTaskProjector:110`、`A2aRemoteAgentOutboundAdapter:89`、`VersatileClient:128`)。第一遍只列了 2 处(`A2aAgentExecutor:167/403`)。

### 维度 C(架构/治理)—— **增强最显著的维度**
11. **🔴 幽灵也在 `architecture-status.yaml` 里(第二遍 H2)** — **这是最关键的增强**。第一遍把 status.yaml 当成"唯一对瘦身内部诚实的文档"并表扬;第二遍发现它的 `:413 allowed_claim`(治理体系最高信任的"已交付"声明面)白纸黑字写着"POST /v1/runs/{runId}/cancel 已 shipped 在 agent-service RunController.java#cancel"——而该文件全仓不存在。**第一遍表扬的文档,第二遍证明它也带幽灵。**
12. **🆕 三条门禁规则被证明"空虚通过"(第二遍 M2/M3/M4)** — 第一遍只点了 Rule 17/85。第二遍逐条拆:
    - Rule 104(`openapi_implemented_route_catalog_truth`)对"目录标已交付但路由不存在"**短路通过**(`_r104_live==0 && continue`)——只守一个方向,对更危险的方向(幽灵)盲。
    - Rule 129(`contract_spi_count_truth`)被**对称幽灵骗过**:agent-bus 多报 3 个 SPI,总数仍凑成 12,算术校验过。
    - Rule 17 单向 + 硬编码 7 名、已陈旧。
13. **🆕 把"被退役的漂移规则"和"它本该抓的发现"一一对上(第二遍 L1)** — 列出 batch 退役的 Rule 134(no_orphan_artefacts)、65(module_metadata_pom_dep_parity)、85、66、135,并指出:正是它们的缺席直接放过了孤儿模块(H1)、版本漂移(M5)、目录↔源 SPI 不一致(H3)。门禁从 ~118 缩到 32,伤亡集中在漂移检测。结论比第一遍重:**"绿"很大程度是因为抓漂移的规则被主动撤了。**
14. **🆕 BoM 自引用版本会解析失败(第二遍 M5)** — `spring-ai-ascend-dependencies` 自引 BoM 写 `0.1.0-SNAPSHOT`,自身已是 `0.2.0-SNAPSHOT`,会解析不到。第一遍的 C-M2 只笼统说"版本漂移"。

### 维度 D(构建/依赖)
15. **🔴 所有 example pom 全部构建不了(第二遍 H3)** — 17 个 example/tool pom 的 parent 写 `0.1.0-SNAPSHOT`,而 parent 已是 `0.2.0-SNAPSHOT` → relativePath 版本不符 + 仓里无此 parent → **没有一个 shipped 示例能对当前 main 构建**。第一遍只说"examples 不在 reactor、CI 不编译"(D-M5),**没发现它们本身已经版本损坏**。对"示例是验证适配器对真实 wire 格式"的产品,这条很重。
16. **➕ openJiuwen 无仓证据更硬(第二遍 H1)** — 第二遍读了 `~/.m2/.../_remote.repositories`,证明 central 下载**失败过**、jar 是手工 `install` 进去的,并引了 travel pom 里中文"需先在 agent-core-java 仓执行 mvn install"的注释。第一遍结论同,但取证更实。
17. **🆕 Spring Boot 4 线程不安全的连锁(第二遍 M3)** — pom/ci 注释自承 `SpringApplication.run()` 非线程安全,CI 因此被迫去掉 `-T 1C` 并加 900s fork-kill 兜底,**限制了 CI 并行度**。第一遍 D-M3 只说"钉在非 GA 里程碑"。

## 三、⚠️ 打架:两遍结论相反(必须裁决,别两份都信)

### 打架点 1 — 提交的 LLM key 严重度:🔴HIGH(第一遍) vs 🟢LOW(第二遍)
- **第一遍 B-H1**:`sk-x00550472` 是"真实形状的 key、提交进 git、若曾真用过则永久泄漏" → **HIGH**。
- **第二遍**:同一行,判定为"**env 可覆盖的占位符**(`${SAA_SAMPLE_LLM_API_KEY:sk-x00550472}`)、仅在 examples、**非 live secret**,只是 `x00550472` 形似 Huawei W3 工号" → **LOW**。
- **裁决倾向**:第二遍的读法在机制上更准(它确实是 `${ENV:default}` 占位符,不是裸 key)。但"形似真实工号"的顾虑两遍都在 → **动作不变(清掉/换成明显占位符),但紧迫度从 P1 降到卫生级**。第一遍此处偏过判。

### 打架点 2 — 取消竞态是否存在:存在(第一遍 A-M1) vs 不存在(第二遍)
- **第一遍 A-M1**:`finally{inFlight.remove}` 在 `inFlight.put` 之前,落在窗口的 cancel 读到 null、不置标志不拆流,执行线程仍发出站调用并对已 CANCELED task 发射 → **MEDIUM 真缺陷**。
- **第二遍**:明确说取消链路**正确**——"in-flight 注册发生在 `handler.execute` 之前、连接窗口有空槽占位、每个分支都重查 cancelled 标志"。
- **裁决**:**两者直接矛盾,必有一错。** 这是最该人工复核的一条——需要回到 `A2aAgentExecutor.java:236-264,285-297,336` 把 put/remove 与 cancel 读取的真实时序对齐确认。在裁决前,不应把 A-M1 当已确认缺陷,也不应当已澄清的非问题。

## 五、给你的一页结论

1. **四个 HIGH 大簇两遍独立复现 → 修复清单按这四块走是稳的**(Versatile 泄漏 / openJiuwen 不可复现 / 幽灵契约 / 租户靠不存在的网关)。
2. **第二遍的真增量价值在两处**:
   - 把"幽灵契约"从契约目录追进了 **status.yaml 的 allowed_claim**(最高信任面),并把 **3 条门禁规则证明为空虚通过**——这把第一遍"治理绿是侥幸"的判断从观点坐实成了机制。
   - 把 **example 全线构建损坏(H3)** 和 **超时→假 COMPLETED(H3)** 这两条第一遍没有的发现补上,前者影响"示例可信度",后者是数据完整性级。
3. **两处打架必须裁决**:提交 key(倾向第二遍 LOW)、取消竞态(必须回代码对时序,二者必有一错)。
4. **第二遍不是超集**:它漏了第一遍的错误体泄漏(A-H3)和远程腿租户缺口(B-M1)。**所以最完整的结论 = 两遍取并集 + 对两处打架做裁决**,而不是用任一单遍替代。

> **方法学回收**:这次重跑用实测证明了"单次评审会漏"——两遍在四个大问题上收敛(可信),在中低层尾部发散(各有 ~3 条对方没有),并且第二遍能纠正第一遍的一处过判、挖出第一遍表扬过的文档里的幽灵。这正是"对抗式 / 多遍验证"在工程评审里的实际收益形态。
