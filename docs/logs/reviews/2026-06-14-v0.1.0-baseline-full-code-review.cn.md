## 一、总体结论(执行摘要)

v0.1.0 基线的**核心协议代码(A2A ingress)质量相当高、并发设计克制成熟**,这是值得肯定的工程底子。但作为一个**面向银行客户的多租户平台的"基线版本"**,它目前有三类系统性问题必须在 0.2.0 周期内正视:

| 维度 | 一句话结论 | 最高严重度 |
|---|---|---|
| **正确性/并发** | 核心 A2A 执行器很扎实;**新引入的 Versatile 出站客户端是明显短板**(线程+连接泄漏) | 🔴 HIGH |
| **安全/租户** | 隔离完全依赖一个"尚不存在的前置网关";**出口侧凭证/日志卫生是最薄弱的具体面** | 🔴 HIGH |
| **架构/治理一致性** | **"幽灵契约"问题在基线里活着**——契约目录承诺的东西代码里没有;瘦身留下悬空守护和孤儿模块 | 🔴 HIGH |
| **构建/依赖/测试** | **干净克隆无法离线构建**(openJiuwen 依赖无公开仓/未 vendor);CI 诚实但质量插件不拦截 | 🔴 HIGH |

**最该优先处理的 3 件事**(详见第三节):
1. **可复现性**:`com.openjiuwen:agent-core-java` 没有公开仓、未 vendor,干净环境编译不了核心模块——CI 目前是靠预置缓存"侥幸绿"。
2. **幽灵契约清理**:契约目录把不存在的 `/v1/runs` REST 接口、不存在的 agent-service 实现、不存在的 3 个 SPI 标成"已交付"——这正是本项目自己最警惕的失败模式,在基线里复发了。
3. **租户隔离的真实性**:整个隔离假设建立在"有一个前置鉴权网关剥离并重注入 `X-Tenant-Id`"之上,而基线里这个网关不存在(agent-service 是空壳),直连 `/a2a` 即可冒充任意租户。

## 二、分维度详细发现

### 维度 A:正确性 / 并发 / 资源安全(agent-runtime)

> 评审者结论:A2A ingress 核心**异常克制、值得肯定**;最弱的是新的 Versatile 出站客户端。

**🔴 HIGH**

- **A-H1 — Versatile 客户端在消费者提前停止时泄漏虚拟线程 + HTTP 连接,生产者永久阻塞**
  `engine/versatile/VersatileClient.java:106-167`。返回的 Stream 没有 `onClose` 钩子,生产者虚拟线程用阻塞式 `queue.put` 写 256 槽队列;消费者一旦提前停止(`VersatileStreamAdapter` 遇终态即 `takeWhile` 停拉、或取消时关流),生产者读满队列后**永久阻塞在 put**,HTTP socket 也被钉死。高负载下每个长/被取消的 Versatile 调用泄漏一个线程+一个连接——而这正是金融场景的热路径。
  *修法方向*:给返回流加 `onClose` 通知生产者停止(取消标志 + 带超时的 offer/丢弃),从生产者侧关闭 `response.body()`。

- **A-H2 — Versatile 客户端的 per-instance 线程池从不关闭**
  `VersatileClient.java:57-63`:`HttpClient` 用 `Executors.newCachedThreadPool()` 构造但该池从无引用、从不 shutdown,类也没有 `close()`/生命周期。多实例时每个泄漏一个缓存池;即便单例,非守护空闲线程也会拖延 JVM 退出。

- **A-H3 — Versatile HTTP 错误体 InputStream 读取后从不关闭**
  `VersatileClient.java:114-121`:4xx/5xx 时 `readAllBytes()` 读错误体后直接 return,未进 try-with-resources,每次上游报错泄漏一个连接。(对比 `AgentScopeRuntimeClient` 显式关了错误体——同仓内有正确范例。)

**🟡 MEDIUM**

- **A-M1 — 取消竞态**:`consumeHandler` 的 `finally{inFlight.remove}` 在 `runRemoteSegment` 的 `inFlight.put` 之前执行;落在该窗口的 cancel 读到 null、既不置取消标志也不拆流,随后执行线程仍发起出站远程调用并对已 CANCELED 的 task 发射——正是注释声称要防止的情况。(`A2aAgentExecutor.java:236-264,285-297,336`)
- **A-M2 — 轨迹 sink 的 `onClose` 与 `accept` 跨线程竞态**:`StampingTrajectoryEmitter.emit` 是 synchronized(因为 openJiuwen 流式在 worker 线程发事件),但 `OtelSpanSink.onClose()` 在执行线程的 finally 里调用、未走同一把锁,而 `openSpans` 是普通 `HashMap`——worker 线程仍在 emit 时即非同步并发访问 HashMap。
- **A-M3 — 缓存的出站 A2A transport(HTTP client)在 shutdown 时不关闭**:`A2aRemoteAgentOutboundAdapter` 的 `transportCache` 仅在端点变更时关旧的,无 bean 生命周期在 context 关闭时清理。
- **A-M4 — Versatile 是 per-poll 超时而非整体 deadline**:`queue.poll(timeoutMs)` 每收一行就重置,缓慢匀速的上游(每次刚好不超时发一行)能让 task 永久 WORKING。连接超时还被复用成读超时,混淆了两种预算。

**🟢 LOW**:A-L1 刷新器把临时中断当永久关闭;A-L2 jitter 可能算出过小延迟无下限钳制;A-L3 排空宽限期硬编码 10s(LLM/远程调用常超),建议改成可配置。

**模块健康度**:A2A ingress 核心(executor/router/trajectory-support/parent-projector)并发纪律优秀——per-task 状态正确保持局部、终态发射延迟到轨迹落地后、append/replace 语义在远程续跑场景处理细致、就绪门 + SmartLifecycle 阶段顺序正确。**唯一明显薄弱区是 Versatile 出站客户端**,应在投产前提升到另两个适配器的标准。

### 维度 B:安全 / 租户隔离(银行多租户视角)

> 评审者结论:隔离全靠一个**尚不存在的前置网关**;出口侧卫生是最薄弱的具体面;核心协议管道防御性编码到位。

**🔴 HIGH**

- **B-H1 — 提交了真实形状的 LLM API key 作为配置默认值**
  `examples/agent-runtime-a2a-versatile-parent-e2e/src/main/resources/application-main.yaml:23`:`api-key: ${SAA_SAMPLE_LLM_API_KEY:sk-x00550472}`。fallback `sk-x00550472` 是 OpenAI-key 形状的字面量,提交进了 git(同类兄弟文件用的是明显占位符)。若曾是真 key 则永久泄漏在历史里。
  *修法*:清成占位符、若曾真用过则轮换、CI 加 secret 扫描。

- **B-H2 — 完整请求载荷(用户原文 + 全部 metadata)在 INFO 级别打日志**
  `A2aAgentExecutor.java:167,403`:`LOG.info(... inputText={} ...)` 和 `... metadata={}`。多租户银行平台把**原始用户消息体 + 整个客户端可控 metadata 映射**写进默认开启的 INFO 日志,metadata 常带业务身份(userId/wap_userName/intent)甚至调用方注入的 secret。违反银行数据处理预期。
  *修法*:INFO 去掉 `inputText=`/`metadata=`(保留 `textChars`/`metadataKeys`),原始载荷降到 DEBUG/TRACE 或走已有的 `TrajectoryMasking`。

**🟡 MEDIUM**

- **B-M1 — 调用方 message metadata 原样转发给远程 agent;跨远程工具边界丢失租户作用域**:入站客户端塞进 metadata 的任何东西被原封不动转发给运营方配置的远程 agent(可能泄漏注入的凭证类字段),且解析出的 `X-Tenant-Id` **没有传播**到远程腿——远程 agent 无法对源自此处的调用做租户隔离。(`A2aRemoteInvocationOrchestrator.java:195-203` → `A2aRemoteAgentOutboundAdapter.java:264`)
- **B-M2 — `ssl-verify: false` 作为出厂默认**:`application-main.yaml:26` 和 openJiuwen 示例配置默认关闭上游模型连接的 TLS 证书校验。示例是银行集成方照抄的模板,默认关 TLS 等于邀请 MITM(而这条链路带着 prompt + api-key Bearer)。

**🟢 LOW**:B-L1 远程 agent 工具用全开放 schema(`additionalProperties:true`)叠加 M1 verbatim 转发,prompt 注入可塞任意字段下行(因端点是运营方配置,非 SSRF,故降级);B-L2 JSON-RPC 错误响应回显原始异常消息,利于侦察。

**记录在案的设计假设(非"已交付漏洞")**:
- **租户信任模型已显式文档化且自洽**:`A2aAgentExecutor.TENANT_STATE_KEY:44-52` 明确写明运行时**不认证**租户——`X-Tenant-Id` 仅在"前置网关剥离客户端头并在鉴权后重注入"的前提下可信。解析优先级正确(头派生 > 客户端自报)。**但基线里 agent-service 是空壳,没有任何前置鉴权/JWT 过滤器,这个网关不存在**——直连 `/a2a` 的客户端可自选租户。
- **远程端点是运营方配置,非客户端可控**:远程只来自 `agent-runtime.remoteAgents[].url`,LLM 只能选"调用哪个已配置 agent",不能选 URL——**无经典 SSRF**。
- **敏感头脱敏存在**:`VersatileClient.SENSITIVE_HEADERS` + `TrajectoryMasking` 对日志/轨迹里的 auth/token/cookie 脱敏;header 透传双路径白名单强制。

**安全姿态**:① 隔离完全建立在一个未构建的边界上——这是银行场景的首要风险;② 出口卫生是最薄弱的具体面(提交的 key、默认关 TLS、verbatim metadata 转发、INFO 打原始载荷);③ 核心协议管道防御性编码到位(无 SSRF、有脱敏、白名单、错误降级、租户优先级正确),所以修复多为"默认值加固 + 日志/转发纪律",而非重构。

### 维度 C:架构 / 治理一致性(漂移检测)

> 评审者结论:法律层**在最该守的地方没守住**;瘦身留下自相矛盾的架构记录、悬空守护、孤儿模块。**项目自己最警惕的"幽灵契约"失败模式在基线里活着。**

**🔴 HIGH**

- **C-H1 — 契约目录 §1 的"已交付 W1 路由"是幽灵:代码里根本没有 `/v1/runs` REST 接口**
  `docs/contracts/contract-catalog.md:21` 把 `POST /v1/runs`、`GET /v1/runs/{id}`、`POST /v1/runs/{id}/cancel` 标为**已交付**,实现指向 `RunController.java`——该文件全仓不存在,无任何 `/v1/runs` 映射,无 `TaskCursor` 类型。真实 HTTP 控制器只有 `A2aJsonRpcController`(A2A JSON-RPC)。目录自称"运行时承诺面",其旗舰承诺不存在。

- **C-H2 — 目录 SPI 表把不存在的实现标成 `shipped`**:`contract-catalog.md:41-45` 把 `Checkpointer/Orchestrator/EnginePort/DefinitionResolver/S2cCallbackTransport` 标"已交付,参考实现 in agent-service",但 `InMemoryCheckpointer/SyncOrchestrator/CompositeDefinitionResolver/InMemoryS2cCallbackTransport` 四个实现文件全不存在,agent-service 是单文件空壳。接口在 agent-bus 存在,但"已交付实现"是假的。

- **C-H3 — 目录称 agent-bus 有 8 个 SPI,其中 3 个不存在**:`contract-catalog.md:87` 列了 `IngressGateway/ReflectionEnvelopeRouter/FederationGateway` 等 8 个,实际只有 5 个(`bus.spi.ingress` 包整个不存在)。目录与 module-metadata 互相矛盾。

- **C-H4 — L0 ARCHITECTURE.md 仍描述 8 模块 reactor,实际是 4**:`architecture/docs/L0/ARCHITECTURE.md:114-116` 标题"Eight-module"并点名 AgentClient/AgentMiddleware/AgentEvolve 等已退役模块。同一文件下方表格(L0:158-164)却又是正确的 4 模块——**文档自相矛盾**。

**🟡 MEDIUM**

- **C-M1 — ArchUnit `RunRepositorySaveGuardTest` 守护的文件已不存在(空虚通过)**:它扫的 `RunController.java`/`SyncOrchestrator`/`InMemoryRunRegistry` 路径在 agent-runtime 里都不存在,扫到 0 个匹配 → 断言空集 → 通过——守护的是一个已删除的设计。
- **C-M2 — 版本号全仓漂移**:根 pom `0.2.0-SNAPSHOT`、agent-bus/runtime metadata `0.0.1-SNAPSHOT`、agent-service metadata `0.1.0-SNAPSHOT`、目录头 `0.1.0-SNAPSHOT`、status.yaml 还引 `rc55`——一个 commit 上四种版本串。
- **C-M3 — `agent-sdk/` 是孤儿模块**:44 个 git 跟踪文件 + 真 pom(parent 写死 `0.1.0-SNAPSHOT`,根已是 0.2.0),却**不在 `<modules>` 里**,只被同样孤儿的 `examples/agent-sdk-example` 引用。永远不在 reactor 构建,独立构建会因 parent 版本不符失败。
- **C-M4 — 门禁唯一的模块对等规则(Rule 106)检测不到这个孤儿**:它比对 pom 模块数(4)和 metadata 文件数(4,agent-sdk 没 metadata 所以不计)——两边都是 4 故通过,而 44 文件孤儿就在磁盘上。没有"每个带 pom 的顶层目录必须在 modules 里或显式豁免"的规则。

**🟢 LOW**:C-L5 Rule 17 是 7 个 SPI 名的硬编码 presence 检查、不验证计数也不拒绝幽灵,所以 H2/H3 的幽灵都能过(而真正能抓这个的 Rule 85 已被"过度耦合"为由退役);C-L6 `discovery/`、`dist/skills/` 是跟踪的生成物但无再生成门禁。

**治理健康度**:① 法律层在最该守的地方没守住——目录(自称"运行时承诺面")有至少三簇独立幽灵契约,而专门抓这个的 Rule 85 被主动退役、幸存的 Rule 17 是 7 名 presence 检查、幽灵都能过;② 架构记录自相矛盾(L0 同文件 8 模块 vs 4 模块);**唯一对瘦身内部诚实的是 `architecture-status.yaml`(明确 reactor_modules:4 + 退役清单),但它不是开发者第一眼看的文档**;③ 瘦身留下悬空守护(空虚通过的 ArchUnit)+ 门禁结构性盲区(孤儿模块逃过 Rule 106)。**门禁能跑、是绿的,但这里的"绿"很大程度是"被退役/弱化的规则不再看了",不是"瘦身很干净"。**

### 维度 D:构建 / 依赖 / 测试质量

> 评审者结论:可复现性是头号缺口;CI 诚实但质量插件不拦截、无 e2e 锚;测试有处即真,覆盖形状不均。

**🔴 HIGH**

- **D-H1 — openJiuwen `agent-core-java` 干净环境无法解析(无公开/声明仓,未 vendor)**:`agent-runtime/pom.xml:78-83`(`0.1.12-jdk17`)、`agent-sdk/pom.xml`(`0.1.12`)等共 8 处引用;唯一声明的仓是 `spring-milestones`,`com.openjiuwen:*` 不在 Maven Central、无仓条目。它能解析**只因为本机 `~/.m2` 已装**。travel 示例 pom 直言"需先在 agent-core-java 仓执行 mvn install"。**干净 clone 的 CI 或新机器编译不了核心模块**——CI 现在是靠预置缓存侥幸绿。这是单一最大可复现性风险。

**🟡 MEDIUM**

- **D-M1 — openJiuwen 坐标版本漂移**:`0.1.12-jdk17`(runtime) vs `0.1.12`(sdk) vs `0.1.7`(示例),硬编码在各模块、未在 parent 统一。`-jdk17` 变体与普通 `0.1.12` 可能字节码目标不同,同 classpath 混用是隐患。
- **D-M2 — openJiuwen 拉进重型/有风险传递依赖**:pulsar-client、sqlite-jdbc、dashscope、openai-java、bouncycastle。在 agent-runtime 里是 `optional` 故不传播给下游,但仍在 runtime 自己的 classpath、且激活 openJiuwen 适配器的部署会全带上。
- **D-M3 — Spring AI 钉在非 GA 里程碑(`2.0.0-M5`)**:`pom.xml:49` + 发布 BoM。里程碑是移动、API 不稳的目标,而该 BoM 是"对外版本契约",消费者会继承这个里程碑依赖。
- **D-M4 — 质量插件配置成非阻断**:SpotBugs `failOnError=false`、PMD/Checkstyle `failOnViolation=false`——`-Pquality verify` 不会因 findings 失败,CI 的"质量"步只出报告没人被迫看。
- **D-M5 — CI 无行为/E2E 锚**:`examples/` 没有聚合 pom、不在 reactor,`mvn verify` 从不编译/运行它们;operator-shape smoke 门禁被(诚实地)排除。A2A wire 端到端路径在 CI 里没有门禁,行为回归可能绿着落地。

**🟢 LOW**:D-L1 发布 BoM 管理了不存在的 `agent-eval`、漏掉实际发布的 runtime/bus/sdk(三份不一致的模块清单);D-L2 `--strict-checksums` 耦合了里程碑仓的校验和卫生;agent-bus(4/21)和 agent-sdk(5/37)测试覆盖偏薄。

**值得肯定的**:
- ✅ **CI 对 fail-open 是诚实的**——无 `|| true`、无 `continue-on-error`;唯一相关注释是解释为何**故意**排除 smoke 门禁(而非掩盖)。跑全 reactor + 架构图 + baseline + 门禁自测。
- ✅ **测试断言真实行为,非同义反复**;**无 `@Disabled`/`@EnabledIfEnvironmentVariable`/`assumeTrue`**——所以没有"集成测试在 CI 静默跳过"这个常见 fail-open 洞。
- ✅ **ArchUnit 架构测试存在**(RuntimePackageBoundary / SpanTenantAttributeRequired / AudienceBExtensionSeams 等)。
- ✅ **仓库卫生好**:无提交的 target/、无 .DS_Store、无 macOS " 2.java" 副本、无误提交的 vendored jar。
- ✅ **agent-service 0 测试是合理的**(它是真·单文件空壳,无逻辑)。

**构建/测试健康度**:① 可复现性是头号缺口(HIGH)——干净 clone 因 openJiuwen 无仓/未 vendor 而编不了核心;② CI 完整性确实好(无掩盖、无静默跳过、全 reactor + 架构门禁)**但**质量插件全非阻断、无 e2e/smoke 锚;③ 测试有处即真、有 ArchUnit 边界测试,但覆盖形状不均 + openJiuwen 坐标漂移 + 两份 BoM 模块清单不一致。

## 三、按优先级排序的修改建议

> 以下是建议,**不是已执行的改动**。0.2.0 周期的处置由你和团队定。

### P0 — 上线/交付前必须处理(基线作为"对外承诺"的诚信与可用性)

1. **修可复现性(D-H1)**:openJiuwen 依赖三选一——发布到可访问仓并声明 / vendor 进 `third_party/` 配文档化的 `install:install-file` 步骤 / 用 Maven profile 把 openJiuwen 适配器隔离使核心 reactor 能脱它构建。**这条不解决,"干净克隆能构建"就是假的,新同事和 CI 随时会被它绊倒。**
2. **清幽灵契约(C-H1/H2/H3/H4)**:把契约目录的 `/v1/runs` REST 家族、agent-service 的四个不存在实现、agent-bus 的三个不存在 SPI,要么改成 `design_only`、要么删除;L0 §2 的"8 模块"标题改成 4 模块现实。**这是项目自己定义的最严重失败模式,基线带着它对外发布会严重误导接手人和审计。**
3. **租户隔离的真实性表态(B 设计假设 + agent-service 空壳)**:基线没有前置鉴权网关,直连 `/a2a` 可冒充租户。要么在文档/release notes 里**极其醒目**地声明"本基线必须部署在剥离+重注入 X-Tenant-Id 的鉴权网关之后,否则无租户隔离",要么把 JWT 交叉校验过滤器纳入基线。银行客户场景这条不能含糊。

### P1 — 0.2.0 早期处理(真实缺陷,有爆炸半径)
