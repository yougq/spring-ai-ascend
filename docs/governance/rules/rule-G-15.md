# Rule G-15 — Fact-Layer Integrity

## What

Rule G-15 polices the new `architecture/facts/` directory introduced by
ADR-0154 (the "fact layer for AI-unbiased L1 understanding" response to
the 2026-05-27 expert review).

It is the **structural** enforcement layer for the policy decision that
factual claims about code, contracts, tests, dependencies, runtime
behaviour, or verification MUST be generated by deterministic scripts —
not authored by humans, never authored by LLMs.

## Why

The L1 Feature Registry shipped W1-W5 on 2026-05-27 carried hand-authored
factual fields (`saa.devPaths`, `saa.verificationTestFqns`,
`saa.verificationCommands`, `saa.sourceFile`). Hand-authored factual
fields race with the code that owns them and lose: rc5 through rc40
recorded 14+ recurrences of F-numeric-drift, plus F-deleted-module-name-
leakage, F-contract-spi-count-drift, and F-recurring-family-yaml-md-drift.

Rule G-13 (Single-Source Rendering Coherence) closed this gap for
*rendered* surfaces by making the Markdown the output of a deterministic
render. Rule G-15 extends the discipline to *extracted* facts: an
extractor reads code, contracts, tests, build files, and config to emit
JSON facts whose provenance points back to the source authority. AI
agents read those facts as the primary input; the gate keeps the layer
honest.

## Sub-clauses

### .a — Fact directory structure

**Enforcer**: E179 sub-check a. Status: advisory at W1, blocking from W2.

The following MUST exist on disk:

- `architecture/facts/README.md` with the level/view/status front-matter
  declared by ADR-0154.
- `architecture/facts/schema/fact.schema.yaml` parseable as YAML.
- `architecture/facts/generated/` (directory; `.gitkeep` placeholder
  allowed when empty).
- `architecture/profile/saa-property-authority.yaml` parseable as YAML.

The schema MUST declare `schema_version`, `type: object`, and the eight
required provenance fields (`fact_id`, `fact_kind`, `source_kind`,
`source_path`, `extractor`, `extractor_version`, `repo_commit`,
`observed_value`).

### .b — Provenance field presence

**Enforcer**: E179 sub-check b. Status: blocking from Wave 2.

Every entry in every `architecture/facts/generated/*.json` file MUST
carry all eight provenance fields. Optional fields (`source_symbol`,
`source_span`, `source_refs`, `notes`) MAY be absent. The `repo_commit`
field MUST be 40 lowercase hex chars; `extractor` MUST resolve to a
binary id under `tools/architecture-workspace/` or
`gate/lib/extractors/`.

Reproducibility note: `generated_at_utc` is INTENTIONALLY OMITTED. Live
timestamps break Rule G-13.b's byte-identical regeneration contract.
`repo_commit` serves as the SLSA-style provenance anchor.

### .c — Byte-identical regen + LLM-no-author

**Enforcer**: E179 sub-check c. Status: blocking from Wave 4 (after the
first non-trivial extractors land).

For every file under `architecture/facts/generated/`, re-running the
declared extractor at the same workspace HEAD MUST produce
byte-identical output. Manual edits (LLM or human) are detected by the
gate comparing committed content to extractor output.

The on-disk banner `# DO NOT EDIT — generated by <extractor> at commit
<repo_commit>` MUST appear at the top of every generated fact file. The
gate checks for the banner; missing banners fail .c.

LLM-no-author: any AI agent that needs to change a fact MUST change the
extractor binary OR the source authority (code, contract YAML, ADR),
then re-run the extractor. Direct authoring of generated fact files is
gate-rejected.

### .d — FunctionPoint hard-evidence fields

**Enforcer**: E179 sub-check d. Status: advisory at Wave 5, blocking
at Wave 6 after corpus sweep.

A function point in `architecture/features/function-points.dsl` with
`saa.status "shipped"` AND channel `http` or `spi` MUST carry:

- `code_entrypoint_refs[]` non-empty AND every entry resolves against
  `architecture/facts/generated/code-symbols.json` (Wave 4 fact);
- `test_refs[]` non-empty AND every entry resolves against
  `architecture/facts/generated/tests.json`;
- at least one of `input_contract_refs[]` / `output_contract_refs[]`
  non-empty AND every entry resolves against
  `architecture/facts/generated/contract-surfaces.json` (Wave 3).

The remaining 11 fields of the v1 FunctionPoint schema (actor, channel,
trigger, preconditions, error_modes, state_changes, side_effects,
auth_policy_refs, tenant_policy_refs, idempotency_policy_refs,
observability_refs, source_authority_refs) are ADVISORY: `null` is
acceptable until a source authority exists, but a non-null value MUST
resolve back to a fact id OR a named ADR / contract / policy file.

## Relationship to Rule G-13

Rule G-13 (Single-Source Rendering Coherence) polices rendered Markdown
and YAML — files in the `templated` and `hybrid` buckets of
`docs/governance/templates/surface-classification.yaml`. Rule G-15
polices **extracted** facts — files under `architecture/facts/generated/`
that are emitted by an extractor reading code/contracts/tests/build/
config/adr sources.

The two rules share the byte-identical-on-regen invariant but apply it
to different artifact classes. Rule G-15 does NOT duplicate Rule G-13;
the cleanup audit at W6 of THIS plan adds a row to `surface-
classification.yaml` for `architecture/facts/generated/*.json`
classifying it as `templated` with the extractor as the renderer, so
G-13.b also catches drift as defence-in-depth.

## Relationship to Rule G-3 (kernel-card coherence)

Rule G-3 polices the CLAUDE.md kernel ↔ rule-card ↔ deferred-doc
triangle. Rule G-15's kernel paragraph in CLAUDE.md byte-matches the
`kernel:` scalar in this card (sub-clause G-3.b), the heading exists in
CLAUDE.md (sub-clause G-3.c), and the kernel fits within `kernel_cap: 8`
sentences (sub-clause G-3.a). Self-validated by re-running the gate.

## Self-tests

Per Rule 110 META, every prevention rule must declare `scope_surfaces:`
(declared above) AND carry ≥2 self-test fixtures across distinct
surfaces. Rule G-15 declares 6 surfaces; fixtures ship incrementally:

- W1 — Fixture 1 (sub-clause .a): synthetic missing
  `architecture/facts/schema/fact.schema.yaml` → Rule fails. Fixture 2
  (sub-clause .a): synthetic malformed YAML → Rule fails with parse
  error.
- W2 — Fixture 3 (sub-clause .b): synthetic fact file with missing
  `repo_commit` field → Rule fails.
- W4 — Fixture 4 (sub-clause .c): synthetic LLM-author scenario (mutate
  a committed fact file after extraction) → Rule fails on byte diff.
- W6 — Fixture 5 (sub-clause .d): synthetic shipped+http FP with empty
  `code_entrypoint_refs` → Rule fails.

All fixtures live in `gate/test_architecture_sync_gate.sh` following the
existing `test_rule_NNN_*` naming pattern.
